# إدارة الورديات (Shift management)

## نظرة عامة

كل مستخدم لديه صلاحية الكاشير (مدير، كاشير، موظف رفوف، موظف مخبز) يجب أن **يبدأ وردية** قبل إتمام البيع أو تسجيل الاسترجاع. النقد في الدرج يُتتبع عبر **سجل حركات نقدية** مرتبط بالوردية.

## سير العمل

1. **بدء الوردية** (`POST /api/shifts/start`): إدخال النقد عند الافتتاح؛ يُنشأ سجل `cashier_shifts` بحالة `open` وحركة من نوع `opening`.
2. **أثناء الوردية**:
   - كل بيع نقدي يربط `transactions.shift_id` ويُضاف سطر `payment` في `shift_cash_movements`.
   - بيع بالبطاقة يُسجَّل في `transactions` فقط (بدون حركة نقدية).
   - الاسترجاع النقدي يربط `refunds.shift_id` ويُضاف سطر `refund` بمبلغ **سالب**؛ الاسترجاع بالبطاقة بدون حركة نقدية.
3. **تسويات** (اختياري): المحاسب/المدير يمكنهم `POST /api/shifts/:id/adjust` بمبلغ موقّع (إيداع/سحب من الدرج) أثناء الوردية المفتوحة.
4. **إغلاق الوردية** (`POST /api/shifts/:id/end`): إدخال النقد الفعلي عند الإغلاق والملاحظات. النظام يحسب:
   - **المتوقع** = افتتاح + مجموع مبيعات النقد للوردية − مجموع مرتجعات النقد للوردية + مجموع حركات `adjustment`.
   - **الفرق (variance)** = النقد عند الإغلاق − المتوقع (موجب = زيادة، سالب = عجز).

## الأدوار

| الدور | بدء/إغلاق وردية | تدقيق كل الورديات |
|--------|------------------|-------------------|
| كاشير / رفوف / مخبز / مدير | نعم (خاصة بهم) | لا (المدير يرى التقارير المالية + يمكنه إغلاق وردية عامل آخر) |
| محاسب | لا (لا يستخدم الكاشير عادة) | نعم مع المدير عبر `/shift-audit` |

عرض وردية واحدة: صاحب الوردية، أو مدير/محاسب.

## واجهات API (ملخص)

| الطلب | المسار | ملاحظات |
|--------|--------|---------|
| POST | `/api/shifts/start` | جسم: `{ opening_cash }` |
| GET | `/api/shifts/current` | الوردية المفتوحة للمستخدم الحالي + `transactions_count` |
| POST | `/api/shifts/:id/end` | جسم: `{ closing_cash, notes }`؛ الكاشير صاحب الوردية أو المدير |
| GET | `/api/shifts` | مدير/محاسب؛ فلاتر: `status`, `cashier_id`, `date_from`, `date_to` |
| GET | `/api/shifts/:id` | تفاصيل + مبيعات + مرتجعات + حركات نقدية + ملخص |
| GET | `/api/shifts/:id/cash-movements` | الحركات النقدية فقط |
| GET | `/api/shifts/:id/export.csv` | تصدير CSV (UTF-8 مع BOM) |
| POST | `/api/shifts/:id/adjust` | مدير/محاسب؛ جسم: `{ amount, description }` |

البيع (`POST /api/checkout`) والاسترجاع (`POST /api/refunds`) يتطلبان **وردية مفتوحة** للكاشير.

## قاعدة البيانات

- الجداول: `cashier_shifts`, `shift_cash_movements`.
- أعمدة مضافة: `transactions.shift_id`, `refunds.shift_id`.
- ملف SQL مرجعي: [backend/database/migrations/001_cashier_shifts.sql](backend/database/migrations/001_cashier_shifts.sql)
- عند التشغيل العادي، يطبّق [backend/database/init.js](backend/database/init.js) الهجرة تلقائياً.

## استكشاف الأخطاء

- **لا توجد وردية مفتوحة** عند البيع: ابدأ وردية من شاشة الكاشير (نافذة الحجب).
- **تعذّر الاتصال بالخادم**: تأكد أن الـ API يعمل على المنفذ 5000 في التطوير، أو ضبط `REACT_APP_API_BASE`.
- **قاعدة بيانات قديمة**: شغّل التطبيق مرة واحدة ليُنفَّذ `init.js`، أو طبّق SQL يدوياً ثم أضف أعمدة `shift_id` إن لزم.

## واجهة المستخدم

- الكاشير: حجب حتى بدء الوردية؛ رأس الصفحة يعرض بداية الوردية وعدد المبيعات؛ زر إغلاق الوردية.
- التدقيق: `/shift-audit` من التقرير اليومي (مدير/محاسب) أو من شريط الكاشير (مدير).

## عتبة التحذير

في الواجهة، إذا تجاوز |الفرق| **100** (نفس وحدة العرض الحالية للأسعار) يظهر تحذير قبل/عند إغلاق الوردية. الثابت: `SHIFT_VARIANCE_WARNING` في [frontend/src/components/ShiftEnd.jsx](frontend/src/components/ShiftEnd.jsx).
